# Unit Test Case Generator

输出可直接落地的单测方案与示例测试代码，默认采用“输入 -> 执行 -> 输出断言”风格，不把测试绑定到内部实现细节。

## Workflow

1. 先识别测试对象和可观察输出
- 列出函数/模块的输入、返回值、副作用、异常行为。
- 明确哪些结果可由外部观察到；仅对这些行为写断言。

2. 再建立测试矩阵
- 必须覆盖 6 类 case：正常、边界、非法、冲突、回归、组合。
- 每类至少 1 条；高风险逻辑每类至少 2 条。

3. 再写测试骨架
- 每条测试保持三段结构：`Arrange(输入)` -> `Act(执行)` -> `Assert(结果)`。
- 断言业务结果，不断言私有步骤、内部调用次数、局部中间变量。

4. 最后做可维护性检查
- 测试名描述“前置条件 + 预期行为”。
- 失败信息要能定位问题。
- 避免因重构导致大面积误报。

## Core Rules

1. 只测输入输出，不绑死内部实现
- 使用公开接口触发行为。
- 只断言返回值、状态变化、错误类型、可观察副作用。
- 不断言内部函数调用顺序、私有流程细节、临时变量值。

2. 边界条件优先级高于正常流程
- 强制覆盖：`null/undefined`、空数组、单元素、超长输入、极小/极大值、顺序错乱、重复数据、非法时间范围。

3. 异常路径必须覆盖
- 强制覆盖：参数非法、缺字段、前置条件不满足、冲突不可解、外部依赖失败时的降级/兜底策略。

4. 测试必须长期可维护
- 重构后应尽量保持稳定。
- 新人可读、失败可定位、误报率可控。

## Six-Case Template

对每个目标函数至少输出以下 6 类 case：

1. 正常输入
- 验证主流程结果正确。

2. 边界输入
- 例如：空数组、单元素、最大/最小值。

3. 非法输入
- 例如：缺字段、格式错误、时间倒置。

4. 冲突输入
- 例如：时间重叠、锚点冲突、不可插入时间段、压缩后仍不可行。

5. 回归用例
- 收录真实历史 bug；修复后必须补测试并固定为回归样本。

6. 组合场景
- 多约束同时出现时，验证系统仍满足预期。

## Output Contract

默认按以下结构输出结果，避免只给零散片段：

1. `测试对象与行为摘要`
2. `测试矩阵（6类 case）`
3. `示例测试代码（AAA 结构）`
4. `风险与缺口（未覆盖项）`

如果用户要求“只补 case 不写代码”，仅输出第 1、2、4 部分。

## Guardrails

- 优先复用项目现有测试框架与断言风格，不引入新依赖。
- 若需求与当前实现冲突，先显式写出假设再生成测试。
- 若无法判断期望行为，先产出待确认清单，避免凭空编造断言。
